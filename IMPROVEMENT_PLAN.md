---

## Как работать с планом

- **Из любой сессии:** взять любую задачу со статусом `- [ ]` и не помеченную `🔒 in-progress`. Перед началом — поменять статус на `🔒 in-progress @имя/дата`, после — на `- [x] @имя/дата`.
- **Параллельная работа:** задачи помечены пометкой зависимостей `(зависит от: …)`. Если зависимости нет — можно брать в любой момент.
- **Критерий готовности:** каждая задача содержит блок «✅ Проверка» — однозначный способ убедиться, что сделано.
- **Коммитить:** после каждой задачи делать отдельный git-коммит с упоминанием ID задачи (например, `Ю1: убрать Telegram-токен из клиентского JS`).

# 🟥 Юридический блок

## Ю1. Убрать Telegram-токен из клиентского JS
**Тип:** безопасность · **Зависимости:** нет
**Файлы:** [src/components/Contact.astro](src/components/Contact.astro), [.env.example](.env.example)

**Проблема:** переменная `PUBLIC_TELEGRAM_BOT_TOKEN` в Astro попадает в клиентский бандл — токен бота виден всем через DevTools. Любой посетитель может от вашего имени читать чат и спамить.

**Шаги:**
1. Сменить токен бота через @BotFather (старый скомпрометирован, если сайт уже деплоился).
2. Выбрать один из вариантов отправки заявки:
   - **Вариант A (проще всего):** заменить отправку на форму через [Web3Forms](https://web3forms.com/) или [Formspree](https://formspree.io/) — пишут на email, токенов в коде нет.
   - **Вариант B:** перевести Astro в режим SSR, создать `src/pages/api/contact.ts` с серверным `fetch` к Telegram API; токен — только в серверной env-переменной без префикса `PUBLIC_`.
   - **Вариант C:** Cloudflare Worker / Netlify Function как прокси для Telegram API.
3. Удалить переменные `PUBLIC_TELEGRAM_BOT_TOKEN` и `PUBLIC_TELEGRAM_CHAT_ID` из `.env.example` и из `.env` (заменить на серверные `TELEGRAM_BOT_TOKEN` без префикса).
4. Проверить, что в собранном `dist/` нет строки `bot<токен>`.

**✅ Проверка:**
- В DevTools → Network → JS-файлах после открытия сайта не находится подстрока с реальным токеном бота.
- Заявка с формы доходит (проверить тестовой отправкой).

## Ю8. Перенос хостинга на белорусский
**Тип:** юр + SEO · **Зависимости:** Ю6 · **Действие пользователя**

**Проблема:** GitHub Pages — зарубежный хостинг. Для коммерческих ресурсов в РБ это серая зона, а после регистрации в БелГИЭ возможны вопросы.

**Шаги:**
1. Выбрать хостера: [hoster.by](https://hoster.by/), [besthost.by](https://besthost.by/), [bn.by](https://bn.by/).
2. Выбрать тариф под статичный сайт (хватит самого дешёвого, ~5-10 BYN/мес).
3. Собрать сайт командой `npm run build`, загрузить содержимое `dist/` на хостинг через FTP/панель.
4. Привязать домен `.by` к новому хостингу.
5. В [astro.config.mjs](astro.config.mjs) поменять `site` на новый домен и убрать `base: '/atelierderose'` (если домен корневой).

**✅ Проверка:**
- Сайт открывается на новом домене.
- В whois IP-адреса домена видно белорусского провайдера.
- Старый адрес GitHub Pages редиректит на новый домен (или закрыт).

# 🟧 SEO-блок (критичный)

## С1. Унификация канонического домена
**Тип:** SEO · **Зависимости:** Ю6 (нужен реальный домен)
**Файлы:** [astro.config.mjs](astro.config.mjs), [public/robots.txt](public/robots.txt), [src/components/StructuredData.astro](src/components/StructuredData.astro)

**Проблема:** сейчас в коде три разных URL: `antaninavaloshchanka.github.io/atelierderose` (config), `atelierderose.by` (StructuredData, robots.txt). Поисковики не понимают, какой канонический.

**Шаги:**
1. В `astro.config.mjs` поставить `site: 'https://<новый-домен>'` и убрать/обновить `base`.
2. В `public/robots.txt` поменять `Sitemap:` на `https://<новый-домен>/sitemap-index.xml`.
3. В `src/components/StructuredData.astro` использовать тот же домен (можно через переменную из `site.json` или через `Astro.site`).
4. В `src/data/site.json` добавить поле `url` и читать его везде.

**✅ Проверка:**
- `curl -s https://<домен>/robots.txt` возвращает корректную ссылку на sitemap.
- `view-source:<домен>` → `<link rel="canonical">` указывает на тот же домен.
- В JSON-LD скрипте `"url": "https://<домен>"` совпадает.
